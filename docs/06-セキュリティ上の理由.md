# セキュリティ上の理由

> **English version:** [06-security-rationale.md](06-security-rationale.md)

本ドキュメントでは、各ハードニング項目を採用した理由をまとめています。設定の背景を理解することで、将来的な運用や調整が容易になります。

## SSH ハードニング
- **root ログイン無効化**：最も権限の高いアカウントを直接攻撃されるリスクを排除  
- **パスワード認証の無効化**：総当たり攻撃やパスワード漏洩のリスクを軽減  
- **鍵認証の強制**：より強固で安全な認証方式  
- **パーミッション強化**：鍵ファイルの不正変更や漏洩を防止  

## ファイアウォール制御
- **firewalld の有効化**：動的で管理しやすいファイアウォールレイヤーを提供  
- **公開サービスの最小化**：攻撃対象領域を削減  
- **デフォルト拒否ポリシー**：意図しないサービス公開を防止  

## システムパラメータの強化
- **カーネル情報の制限**：メモリ情報やポインタ情報の漏洩を防止  
- **逆方向パスフィルタ**：IP スプーフィング対策  
- **ネットワークスタックの調整**：特定のネットワーク攻撃に対する耐性向上  

## 監査ログ
- **監査ルールの拡張**：認証・設定変更など重要操作の可視化  
- **永続的なログ管理**：再起動後もログが保持される  
- **アカウンタビリティの確保**：インシデント対応やコンプライアンスに有用  

## 運用上の考慮点
- セキュリティと可用性のバランスが重要  
- 各変更は検証を行いながら段階的に適用  
- 文書化により再現性と保守性を確保  

これらの理由により、本ベースラインは安全性と運用性の両立を目指しています。
